## 3. TypeScript optionality rule

The generated `for_main_repo/dtos.ts` applies this rule **uniformly**:

> A property is **mandatory** (`name: T`) iff it carries the
> `[DataMember]` attribute in the C# source. Otherwise it is **optional**
> (`name?: T`).

Rationale:
- The `[DataMember]`-bearing properties are the explicit serialization
  contract — the server WILL emit them and we can rely on them being
  present.
- POCO-style properties are emitted by Newtonsoft's default contract
  resolver, which is sensitive to null-handling settings the host
  binary's config may have changed. Treating them as optional is the
  safe lift to TypeScript-land — if the server *does* omit one (or the
  field's value is `null`), the RN code still type-checks.

This is **not** the same as `IsRequired = true` on `[DataMember]`. The
`IsRequired` flag is decoded from the attribute blob (Phase 2 dropped the
raw bytes in `dtos.json`) — every `[DataMember]` we surveyed carries the
default `IsRequired = false`, so we cannot tighten the TS optionality
beyond what the rule above gives us. **Confidence on the rule: 90%**.

## 4. DateTime handling on the wire

Properties typed `DateTime` (e.g. `Token.CreateDate`) are serialized by
`Newtonsoft.Json v9.0.0.0` with whatever `DateFormatHandling` the host
binary's `JsonSerializerSettings` was configured with. The defaults are:

- `DateFormatHandling.IsoDateFormat` → `"2024-09-15T14:30:00Z"` (since Newtonsoft 4.5+).
- The legacy form `"/Date(unix_ms)/"` was the **default before 4.5**.

We have no .NET-side configuration code in the recoverable C# (the
tampered classes include the config bootstrapping). The `dtos.ts` file
therefore models every DateTime as **`WcfDateTime = string`** — opaque to
the client. If the wire ever turns out to use the legacy `/Date(…)/`
shape, this still type-checks; only the parsing code in the RN client
needs to know.

**Confidence: 70%** that the wire uses ISO-8601. The remediation is
trivial (single helper to parse both shapes), so we did not pursue this
further.
